# NotebookLM MCP — Infographic Generation Specification for Kijko Wiki

> Defines how the WikiAgent uses the NotebookLM MCP server to generate
> infographics at **Detailed** level of detail for wiki pages.

---

## Overview

The NotebookLM MCP server provides zero-hallucination knowledge synthesis
and visual asset generation. For the Kijko wiki system, we use it to generate
infographic diagrams that embed directly into Fumadocs MDX pages.

**Detail Level**: `Detailed` — shows all major components, relationships,
data flows, and key metrics. This is the maximum useful detail for documentation
infographics (higher than "Overview" which only shows top-level boxes).

---

## Infographic Types per Wiki

For each repository wiki generated by the WikiAgent, produce these infographics:

### 1. Architecture Overview Diagram
**Page target**: `concepts/architecture.mdx`
**Content source**: CodeGraph symbol graph + CGC dependency analysis

```
Prompt Pattern:
"Create a detailed architecture infographic for the [REPO_NAME] system.

Show:
- All major modules/packages and their responsibilities
- Data flow arrows between components (labeled with data types)
- External service integrations (databases, APIs, message queues)
- Entry points (HTTP routes, CLI commands, event handlers)
- Deployment boundaries (containers, processes, services)

Style: Clean technical documentation diagram
Color: Dark background (#0f1117) with [brand_accent] highlights
Layout: Horizontal left-to-right data flow
Labels: Component names, protocol annotations, port numbers

Source data:
[PASTE CodeGraph module summary]
[PASTE CGC dependency graph output]"
```

**Fumadocs embed**:
```mdx
<ImageZoom
  src="/docs/[repo]/infographics/architecture-overview.png"
  alt="Architecture overview of [REPO_NAME]"
  width={1200}
  height={675}
/>
```

### 2. Dependency Graph Visualization
**Page target**: `concepts/data-model.mdx` or `reference/dependencies.mdx`
**Content source**: CGC dependency closure analysis

```
Prompt Pattern:
"Create a detailed dependency graph infographic for [REPO_NAME].

Show:
- All internal modules as nodes (sized by complexity score)
- Dependency edges with direction arrows
- External dependencies grouped in a border zone
- Circular dependency warnings highlighted in red
- Complexity hotspots color-coded (green < 10, yellow 10-20, red > 20)

Style: Node-link diagram, organic layout
Color: Dark background, nodes colored by domain area
Layout: Force-directed with domain clustering
Labels: Module names, dependency counts, complexity scores

Source data:
[PASTE CGC dependency closure JSON]"
```

### 3. Data Flow Diagram
**Page target**: `concepts/[primary-concept].mdx`
**Content source**: CodeGraph callers/callees + route analysis

```
Prompt Pattern:
"Create a detailed data flow diagram for [REPO_NAME].

Show:
- User/client request entry point
- Processing pipeline stages (with stage names)
- Data transformations at each stage
- Storage read/write operations
- Response/output generation
- Error handling branches

Style: Swim-lane or pipeline diagram
Color: Dark background, flow arrows in accent color
Layout: Top-to-bottom vertical flow
Labels: Stage names, data type annotations, timing info

Source data:
[PASTE CodeGraph route → handler → service chain analysis]"
```

### 4. Per-Concept Mental Model Diagrams
**Page target**: Each `concepts/[concept].mdx` page
**Content source**: Relevant CodeGraph analysis for that concept

```
Prompt Pattern:
"Create a focused concept diagram for '[CONCEPT_NAME]' in [REPO_NAME].

Show:
- The 2-3 key entities involved in this concept
- How they relate (arrows with relationship labels)
- A concrete example flow (numbered steps)
- Key properties of each entity

Style: Clean, minimal, focused on understanding
Color: Light or dark background (match page theme)
Layout: Centered, balanced
Labels: Entity names, relationship types, example values

Context:
[PASTE relevant code structures and relationships]"
```

### 5. API Surface Map
**Page target**: `api-reference/index.mdx` (API overview page)
**Content source**: OpenAPI spec + route definitions

```
Prompt Pattern:
"Create a detailed API surface map for [REPO_NAME].

Show:
- All API endpoint groups as labeled zones
- Individual endpoints as items within zones (METHOD /path)
- Color-code by HTTP method (GET=green, POST=blue, PUT=yellow, DELETE=red)
- Show authentication requirements per group
- Indicate which endpoints are public vs internal

Style: Organized grid/table visual
Color: Method-based color coding on neutral background
Layout: Grouped by domain (e.g., Users, Documents, Search)
Labels: Method, path, brief description

Source data:
[PASTE OpenAPI endpoint summary]"
```

## Generation Workflow

```
1. Wiki agent completes Phase 1 (Discovery) and Phase 2 (IA)
2. For each infographic type needed:
   a. Assemble source data from CodeGraph/CGC outputs
   b. Fill in the prompt pattern with repo-specific data
   c. Call NotebookLM MCP with:
      - detail_level: "Detailed"
      - output_format: "png" (or "svg" for scalable)
      - dimensions: 1200x675 (landscape) or 675x1200 (portrait)
   d. Save output to wiki-content/docs/[repo]/infographics/
   e. Embed in target page via <ImageZoom> component
3. QA: Verify all infographics render correctly in dark mode
```

## Quality Criteria

- **Readability**: All text in infographics must be legible at 100% zoom
- **Contrast**: Sufficient contrast ratio (4.5:1+) for all labels
- **Dark mode**: All infographics must work on both light and dark backgrounds
- **Brand consistency**: Use Kijko brand colors for accent elements
- **Accuracy**: Every component shown must correspond to actual code structures
- **Density**: Detailed level means showing component internals, not just boxes-and-arrows

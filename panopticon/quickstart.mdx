This guide walks you through a complete Panopticon setup on your local machine. By the end, you will have the Temporal infrastructure running, the HyperVisa video catalog built, and a tool call executing through the durable pipeline. The entire process takes less than five minutes if you already have Docker and Python installed.

## Prerequisites

<Callout type="info" title="Before you begin">
  - **Python 3.12+** installed
  - **Docker** and **Docker Compose** (v2) installed and running
  - **At least one Gemini API key** (Google AI Studio or Vertex AI Express)
  - At least **2 GB of free RAM** for the Temporal infrastructure
  - At least one downstream MCP server registered (e.g., Firecrawl, Exa)
</Callout>

## Setup

<Steps>
  <Step>
    ### Clone the repository and install

    Clone the Panopticon repository and install it as an editable Python package:

    ```bash
    git clone https://github.com/kijko-ai/Panopticon.git
    cd Panopticon
    pip install -e .
    ```

    This installs the `panopticon` CLI command and all Python dependencies: Click, PyYAML, temporalio, mcp, google-genai, Flask, and HyperVisa.

    Verify the installation:

    ```bash
    panopticon --version
    # panopticon, version 0.1.0
    ```
  </Step>

  <Step>
    ### Start the Temporal infrastructure

    Panopticon requires a running Temporal server for durable workflow execution. The included Docker Compose file sets up PostgreSQL, Temporal Server, and the Temporal Web UI:

    ```bash
    docker compose up -d
    ```

    This starts three containers:
    - **PostgreSQL 15** -- Temporal's persistence backend
    - **Temporal Server** -- gRPC frontend on port 7233
    - **Temporal UI** -- Web dashboard on port 8233

    Wait for Temporal to become healthy (about 30 seconds on first start):

    ```bash
    docker compose ps
    ```

    You should see all three services as "Up (healthy)". Open `http://localhost:8233` in your browser to verify the Temporal UI is accessible.
  </Step>

  <Step>
    ### Configure Gemini API keys

    Panopticon uses Gemini for visual catalog search. Set your API key(s) as environment variables:

    <Tabs items={["Single Key", "Multiple Keys", "Mixed Keys"]}>
      <Tab value="Single Key">
        ```bash
        export GEMINI_API_KEY="AIzaSy..."
        ```
      </Tab>
      <Tab value="Multiple Keys">
        ```bash
        # Comma-separated for round-robin rotation
        export GEMINI_API_KEYS="AIzaSy...key1,AIzaSy...key2,AIzaSy...key3"
        ```
      </Tab>
      <Tab value="Mixed Keys">
        ```bash
        # Regular + Vertex AI Express keys
        export GEMINI_API_KEYS="AIzaSy...regular1,AIzaSy...regular2"
        export VERTEX_API_KEYS="vertex-key-1,vertex-key-2"
        ```
      </Tab>
    </Tabs>

    Multiple keys enable automatic failover: if one key hits the rate limit, Panopticon rotates to the next key with zero downtime.
  </Step>

  <Step>
    ### Register MCP servers

    Register the downstream MCP servers that Panopticon will manage. Each server needs a name, a command to start it, and any required arguments:

    ```bash
    # Example: Register Firecrawl MCP server
    panopticon register firecrawl npx -a @anthropic/firecrawl-mcp

    # Example: Register Exa MCP server
    panopticon register exa npx -a @anthropic/exa-mcp

    # Example: Register a custom Python MCP server
    panopticon register my-tools python -a /path/to/my_mcp_server.py
    ```

    Server configurations are stored at `~/.panopticon/servers.yaml`. You can also edit this file directly:

    ```yaml title="~/.panopticon/servers.yaml"
    servers:
      firecrawl:
        command: npx
        args: ["@anthropic/firecrawl-mcp"]
        env:
          FIRECRAWL_API_KEY: "fc-..."
        enabled: true
      exa:
        command: npx
        args: ["@anthropic/exa-mcp"]
        env:
          EXA_API_KEY: "exa-..."
        enabled: true
    ```
  </Step>

  <Step>
    ### Build the tool catalog

    Discover tools from all registered MCP servers and build the HyperVisa video catalog:

    ```bash
    panopticon discover --rebuild
    ```

    This performs four steps:
    1. Starts each registered MCP server temporarily
    2. Sends a JSON-RPC `tools/list` request to fetch schemas
    3. Renders schemas to a 1fps MP4 video using HyperVisa
    4. Uploads the video to the Gemini File API

    Expected output:

    ```text
    Catalog built: 47 tools from 3 servers
    Video: /home/user/.panopticon/catalog.mp4
    Gemini: files/abc123xyz
    ```

    The catalog persists in the Gemini File API until explicitly deleted. You only need to rebuild it when you add or remove MCP servers.
  </Step>

  <Step>
    ### Start the Temporal worker

    The worker process handles all durable workflow execution. Start it in a separate terminal:

    ```bash
    panopticon worker
    ```

    Expected output:

    ```text
    2026-03-29 10:15:00 panopticon.worker INFO Connecting to Temporal at localhost:7233 (ns=default)
    2026-03-29 10:15:00 panopticon.worker INFO Worker started on queue 'panopticon-tasks' with Nexus service
    ```

    Keep this terminal running. The worker must be active for any tool execution to work.
  </Step>

  <Step>
    ### Execute your first tool call

    Now test the complete pipeline. Use the CLI to search for a tool and execute it:

    <Tabs items={["Browse First", "Smart Execute", "Raw Execute"]}>
      <Tab value="Browse First">
        ```bash
        # Step 1: Find tools for web scraping
        panopticon browse "web scraping"

        # Output:
        #   firecrawl.scrape (confidence: 95%)
        #     Scrape a single URL and return its content
        #   exa.get_contents (confidence: 72%)
        #     Get the contents of a URL
        #   firecrawl.crawl (confidence: 68%)
        #     Crawl a website starting from a URL

        # Step 2: Execute the top match
        panopticon exec firecrawl.scrape '{"url":"https://example.com"}'
        ```
      </Tab>
      <Tab value="Smart Execute">
        ```bash
        # Natural language intent -- Panopticon resolves to tool + params
        panopticon ask "fetch the content of example.com"

        # Output:
        # Resolved: firecrawl.scrape (confidence: 95%)
        # Params: {"url": "https://example.com"}
        #
        # Result:
        # { "content": "...", "title": "Example Domain", ... }
        ```
      </Tab>
      <Tab value="Raw Execute">
        ```bash
        # Direct execution with known server.tool notation
        panopticon exec firecrawl.scrape '{"url":"https://example.com"}'

        # Output:
        # { "tool": "firecrawl.scrape", "result": {...}, "ok": true, "duration_ms": 2340 }
        ```
      </Tab>
    </Tabs>
  </Step>
</Steps>

## Verification Checklist

After completing the steps above, verify your setup:

- Temporal containers are running: `docker compose ps` shows all three as "Up (healthy)"
- Temporal UI is accessible: `http://localhost:8233` shows the Temporal dashboard
- Catalog is built: `panopticon health` shows "Catalog: BUILT" with a tool count
- Worker is connected: `panopticon health` shows "Temporal: OK"
- MCP servers are registered: `panopticon health` lists your servers
- Tool execution works: the `panopticon exec` or `panopticon ask` command returns a result

```bash
panopticon health
```

Expected output:

```text
=== Panopticon Health ===

Temporal: OK (localhost:7233)

MCP Servers: 2
  firecrawl: enabled (npx)
  exa: enabled (npx)

Catalog: BUILT
  Tools: 47
  Updated: 2026-03-29T10:15:00
  Gemini: files/abc123xyz

Total registered tools: 47
```

## Optional: Register the Nexus Endpoint

If you want other Temporal namespaces to call Panopticon's tools, register the Nexus endpoint:

```bash
python -m panopticon.temporal.register_nexus
```

This creates a `panopticon` Nexus endpoint in Temporal that routes to the `panopticon-tasks` task queue.

## Optional: Start the REST API

For dashboard integration, start the REST API server:

```bash
panopticon api --port 8043
```

The API is now available at `http://localhost:8043/api/panopticon/health`.

## Optional: MCP Server Mode

To use Panopticon as an MCP server for Claude Code, start it in stdio mode:

```bash
panopticon serve
```

Add this to your Claude Code MCP configuration to give Claude access to all of Panopticon's tools.

## Common Issues

<Accordions>
  <Accordion title="Temporal connection refused">
    The Temporal server takes about 30 seconds to become healthy on first start. Wait for `docker compose ps` to show all services as healthy. If it stays unhealthy, check the logs: `docker compose logs temporal`.
  </Accordion>
  <Accordion title="No Gemini API keys available">
    Set the `GEMINI_API_KEY` or `GEMINI_API_KEYS` environment variable before running any discovery commands. The key must be a valid Google AI Studio API key starting with `AIzaSy`.
  </Accordion>
  <Accordion title="Catalog build fails with 'No schemas available'">
    This means no MCP servers are registered or all registered servers failed to respond. Check `~/.panopticon/servers.yaml` and verify the server commands are correct. You can test a server manually: `npx @anthropic/firecrawl-mcp`.
  </Accordion>
  <Accordion title="Tool execution times out">
    Ensure the Temporal worker is running (`panopticon worker`). Without a worker, workflows are scheduled but never picked up. Also check that the downstream MCP server can start successfully.
  </Accordion>
  <Accordion title="All keys rate-limited error">
    You have hit the RPM limit on all configured Gemini keys. Add more keys to `GEMINI_API_KEYS` or wait for the cooldown period (default 60 seconds). Check key health: `panopticon health` or the `key_pool_health` MCP tool.
  </Accordion>
</Accordions>

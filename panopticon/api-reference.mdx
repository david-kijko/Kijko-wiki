Panopticon exposes three API surfaces: the ingestion API for writing telemetry data, the query API for reading data and metadata, and the management API for configuring runtime behavior. All three surfaces are available over HTTP with JSON payloads, and the ingestion and query surfaces also support gRPC with Protocol Buffer payloads for high-throughput production use cases. This reference documents every endpoint, its parameters, expected payloads, and error responses.

## Base URLs

| Surface    | Default Port | Base Path     | Protocol     |
|-----------|-------------|---------------|-------------|
| Ingestion | 9091        | `/api/v1`     | HTTP, gRPC  |
| Query     | 9090        | `/api/v1`     | HTTP, gRPC  |
| Management| 9090        | `/api/v1/admin` | HTTP      |

All endpoints accept and return `application/json` by default. To use Protocol Buffers over HTTP, set the `Content-Type` header to `application/x-protobuf` and encode the request body accordingly.

## Authentication

<Callout type="info">
Authentication is optional in standalone mode (the default for local development). In production deployments, enable bearer token authentication by setting `server.auth.enabled: true` in the configuration. All requests must then include an `Authorization: Bearer <token>` header.
</Callout>

## Ingestion API

The ingestion API accepts metric data from application services and infrastructure agents. It is designed for high throughput with minimal latency overhead on the sender side.

### POST /api/v1/ingest

Push a batch of metric samples. This is the primary write endpoint.

**Request Body:**

```typescript
interface IngestRequest {
  /** Array of metric samples to ingest */
  samples: Array<{
    /** Metric name (e.g., "http_request_duration_seconds") */
    metric: string;
    /** Key-value label pairs for this sample */
    labels: Record<string, string>;
    /** Unix epoch timestamp in milliseconds */
    timestamp: number;
    /** Numeric measurement value */
    value: number;
  }>;
  /** Identifier for the sending service (used for rate limiting) */
  source: string;
}
```

**Response (200 OK):**

```json
{
  "accepted": 100,
  "rejected": 0
}
```

**Response (400 Bad Request):**

```json
{
  "error": "validation_failed",
  "details": [
    { "index": 3, "field": "metric", "message": "metric name contains invalid characters" },
    { "index": 7, "field": "timestamp", "message": "timestamp is in the future" }
  ]
}
```

**Response (429 Too Many Requests):**

```json
{
  "error": "rate_limited",
  "source": "my-service",
  "limit": 10000,
  "retry_after_ms": 1000
}
```

### POST /api/v1/ingest/batch

Push a large batch of samples with streaming support. Unlike the standard ingest endpoint, this accepts newline-delimited JSON (NDJSON) for efficient bulk imports.

**Request Body** (NDJSON, one sample per line):

```text
{"metric":"cpu_usage","labels":{"host":"web-1"},"timestamp":1711700000000,"value":0.72}
{"metric":"cpu_usage","labels":{"host":"web-2"},"timestamp":1711700000000,"value":0.45}
{"metric":"cpu_usage","labels":{"host":"web-3"},"timestamp":1711700000000,"value":0.88}
```

**Response (200 OK):**

```json
{ "accepted": 3, "rejected": 0, "elapsed_ms": 12 }
```

### GET /api/v1/ingest/health

Health check for the ingestion service.

```json
{ "status": "ok", "uptime_seconds": 3600, "samples_ingested": 1500000 }
```

## Query API

The query API provides read access to stored metric data. It supports instant queries (single point in time), range queries (time series), and metadata queries (label introspection).

### GET /api/v1/query

Execute a range query over stored metric data.

**Query Parameters:**

| Parameter   | Type     | Required | Description                                    |
|------------ |--------- |--------- |----------------------------------------------- |
| `q`         | string   | Yes      | Metric name or query expression                |
| `from`      | number   | Yes      | Start timestamp (Unix ms)                      |
| `to`        | number   | Yes      | End timestamp (Unix ms)                        |
| `step`      | string   | No       | Resolution step (e.g., "1m", "5m"). Default: auto |
| `labels`    | string   | No       | Label filter (e.g., `method=GET,status=200`)   |
| `aggregate` | string   | No       | Aggregation function: `avg`, `sum`, `min`, `max`, `p50`, `p95`, `p99` |

**Response (200 OK):**

```json
{
  "status": "ok",
  "data": {
    "series": [
      {
        "metric": "http_requests_total",
        "labels": { "method": "GET", "status": "200" },
        "samples": [
          { "timestamp": 1711700000000, "value": 1547 },
          { "timestamp": 1711700060000, "value": 1623 },
          { "timestamp": 1711700120000, "value": 1701 }
        ]
      }
    ],
    "stats": {
      "series_scanned": 1,
      "samples_scanned": 180,
      "elapsed_ms": 4
    }
  }
}
```

### GET /api/v1/query/instant

Execute a point-in-time query. Returns the most recent value for each matching series.

**Query Parameters:**

| Parameter | Type   | Required | Description                          |
|---------- |------- |--------- |------------------------------------- |
| `q`       | string | Yes      | Metric name or query expression      |
| `time`    | number | No       | Timestamp (Unix ms). Default: now    |
| `labels`  | string | No       | Label filter                         |

### GET /api/v1/labels

List all label keys present in the stored data.

```json
{
  "status": "ok",
  "data": ["__name__", "env", "host", "method", "path", "service", "status"]
}
```

### GET /api/v1/labels/\{key\}/values

List all values for a specific label key.

```bash
curl -s "http://localhost:9090/api/v1/labels/service/values" | jq .
```

```json
{
  "status": "ok",
  "data": ["api-gateway", "auth-service", "panopticon", "user-service"]
}
```

### GET /api/v1/series

List all metric series matching a label filter.

**Query Parameters:**

| Parameter | Type   | Required | Description                    |
|---------- |------- |--------- |------------------------------- |
| `match`   | string | Yes      | Label matcher expression       |
| `from`    | number | No       | Start of time range (Unix ms)  |
| `to`      | number | No       | End of time range (Unix ms)    |

## Management API

The management API provides runtime configuration and operational endpoints. These endpoints are only available on the query port and require admin authentication in production.

### GET /api/v1/admin/config

Returns the current runtime configuration as JSON.

### PUT /api/v1/admin/config/reload

Triggers a hot reload of the configuration file. Returns the validation result before applying.

```json
{
  "status": "ok",
  "message": "Configuration reloaded",
  "changes": ["scrape.targets[1].interval: 15s -> 30s"]
}
```

### GET /api/v1/admin/targets

Lists all scrape targets and their current status.

```json
{
  "status": "ok",
  "targets": [
    {
      "url": "http://localhost:9090/metrics",
      "state": "up",
      "last_scrape": "2024-03-29T10:15:00Z",
      "scrape_duration_ms": 23,
      "samples_scraped": 147
    }
  ]
}
```

### POST /api/v1/admin/alerts/rules

Create or update an alerting rule.

```json
{
  "name": "high_error_rate",
  "metric": "http_requests_total",
  "labels": { "status": "500" },
  "condition": "rate(5m) > 10",
  "duration": "2m",
  "severity": "critical",
  "notify": ["webhook:ops-channel"]
}
```

### DELETE /api/v1/admin/alerts/rules/\{name\}

Delete an alerting rule by name.

## Error Codes

All API errors follow a consistent format:

```json
{
  "error": "error_code",
  "message": "Human-readable description",
  "details": {}
}
```

| Code                | HTTP Status | Description                              |
|-------------------- |------------ |----------------------------------------- |
| `validation_failed` | 400         | Request payload failed validation        |
| `query_parse_error` | 400         | Query expression could not be parsed     |
| `not_found`         | 404         | Requested resource does not exist        |
| `rate_limited`      | 429         | Per-source rate limit exceeded           |
| `internal_error`    | 500         | Unexpected server error                  |
| `storage_error`     | 503         | Storage backend is unavailable           |

<Callout type="info">
All 5xx errors include a `request_id` field in the response body. Include this ID when reporting issues to help with log correlation.
</Callout>

## Rate Limits

Default rate limits by endpoint:

| Endpoint              | Default Limit           | Configurable Via            |
|---------------------- |------------------------ |---------------------------- |
| `/api/v1/ingest`      | 10,000 samples/sec/source | `server.rate_limit.ingest` |
| `/api/v1/query`       | 100 queries/sec          | `server.rate_limit.query`  |
| `/api/v1/admin/*`     | 10 requests/sec          | `server.rate_limit.admin`  |

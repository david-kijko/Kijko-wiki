Panopticon is the unified monitoring and observability layer that sits across the entire Kijko infrastructure. It collects telemetry from every service in the workspace, aggregates metrics, correlates logs with traces, and exposes a single pane of glass for operators who need to understand what the system is doing at any given moment. The name comes from the architectural concept of a structure where every part can be observed from a single vantage point -- and that maps directly to what this system does: complete visibility without jumping between a dozen different dashboards.

In the Kijko workspace, Panopticon is not just a monitoring add-on. It is a first-class citizen in the repository's seed data, test suites, and agent help text. The label `panopticon` appears in `server/routes.ts` as a seeded repository, in `tests/wikiagent.test.ts` as an intent classification target, and in `apps/agent/src/lib/chat-engine.ts` as one of the example commands surfaced to users. That three-way presence across data, tests, and user-facing guidance makes it a stable reference point in the workspace vocabulary.

## How It Works

Panopticon operates on a pull-and-push hybrid model. Services can push metrics and traces directly using the ingestion API, or Panopticon can scrape Prometheus-compatible endpoints on a configurable interval. The data flows through four stages:

1. **Ingestion** receives raw telemetry via HTTP, gRPC, or the Prometheus scrape loop and normalizes it into the internal time-series format through the Normalizer component.
2. **Processing** runs three parallel pipelines: an aggregation pipeline that computes windowed summaries, an anomaly detector that maintains statistical baselines using EMA and modified z-scores, and an alert evaluator that triggers on data arrival rather than polling.
3. **Storage** persists processed data into a columnar time-series store with three retention tiers (hot at 48h full resolution, warm at 30d with 1-minute downsampling, cold at 13 months with 1-hour downsampling) and an inverted label index for fast queries.
4. **Query** exposes a unified SQL-like query interface with automatic selection between raw and aggregated data, federation proxy for multi-cluster deployments, and REST/gRPC API surfaces.

The key design decision is that ingestion and query are completely decoupled. The write path and the read path operate on separate subsystems with independent resource budgets, so you can ingest data at high throughput without impacting query performance.

## Pipeline Integration

When the 7-agent documentation pipeline runs against Panopticon, the Audit Agent at `apps/agent/src/lib/pipeline/agents/audit-agent.ts` scans for exported symbols and compares them against existing wiki pages. The pipeline's `PipelineOrchestrator` passes Panopticon's repository path through the full agent chain:

- **Audit** identifies documentation gaps by scanning code symbols
- **IA** generates the page taxonomy (index, architecture, quickstart, api-reference, configuration)
- **Writer** produces MDX content grounded in code references
- **Verifier** validates that file paths and function names in the docs match the actual codebase
- **QA** runs snippet verification in Docker sandboxes using the runtime registry in `docs-pipeline.ts`
- **Monitor** tracks page freshness against code changes
- **Publish** writes validated MDX to `wiki-content/docs/panopticon/`

## Why Panopticon Matters

In any infrastructure that grows beyond a handful of services, observability stops being optional and becomes the foundation that every other operational practice depends on. Incident response, capacity planning, deployment validation, and SLO tracking all require reliable telemetry. Panopticon centralizes that telemetry so you do not end up with fragmented monitoring where each team uses a different tool and nobody has the full picture.

- **Unified data model**: Metrics, logs, and traces share a common labeling scheme, so you can correlate a spike in error rate with the specific trace that caused it and the log entries that describe it.
- **Low-latency alerting**: Alert rules evaluate on the ingestion path, not on a polling loop, which means you get notified within seconds of a threshold breach.
- **Retention flexibility**: Different metric types can have different retention windows. High-cardinality debug metrics might retain for 24 hours, while aggregate SLO data retains for 13 months.
- **Query federation**: Dashboards can query across multiple Panopticon instances in a federated setup without requiring a centralized aggregation layer.

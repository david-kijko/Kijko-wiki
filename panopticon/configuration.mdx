Panopticon uses a file-based configuration system with environment variable overrides. Configuration files are stored at `~/.panopticon/` and control MCP server registration, guardrail enforcement, Temporal connectivity, and memory system behavior. This reference documents every configuration option, its type, default value, and operational implications.

## Configuration Directory

All Panopticon configuration lives under `~/.panopticon/`:

```
~/.panopticon/
  servers.yaml           # MCP server registrations
  catalog.json           # HyperVisa video catalog metadata
  schema_cache.json      # Cached tool schemas from MCP servers
  guardrails.yaml        # Smith guardrail enforcement policy
  memory/
    episodic.jsonl       # Episodic memory records
    hive.jsonl           # Hive mind observations
    promotions.jsonl     # Promoted patterns
    traces.jsonl         # Archived execution traces
```

Panopticon creates these directories automatically on first run via the `ensure_dirs()` function in `panopticon/config.py`.

## MCP Server Configuration

MCP servers are registered in `~/.panopticon/servers.yaml`. Each entry defines how to start a downstream MCP server subprocess.

```yaml title="~/.panopticon/servers.yaml"
servers:
  firecrawl:
    command: npx
    args:
      - "@anthropic/firecrawl-mcp"
    env:
      FIRECRAWL_API_KEY: "fc-your-key-here"
    enabled: true

  exa:
    command: npx
    args:
      - "@anthropic/exa-mcp"
    env:
      EXA_API_KEY: "exa-your-key-here"
    enabled: true

  graphiti:
    command: python
    args:
      - "-m"
      - "graphiti_mcp"
    env:
      GRAPHITI_HOST: "localhost"
      GRAPHITI_PORT: "7474"
    enabled: true

  notebooklm:
    command: node
    args:
      - "/path/to/notebooklm-mcp/dist/index.js"
    env: {}
    enabled: false
```

### Server Options

<TypeTable
  type={{
    command: { type: 'string', description: 'Executable to run (e.g., npx, python, node)', required: true },
    args: { type: 'string[]', description: 'Command-line arguments passed to the executable', default: '[]' },
    env: { type: 'Record<string, string>', description: 'Environment variables injected into the subprocess', default: '{}' },

### Registering Servers via CLI

```bash
# Register with arguments and environment variables
panopticon register firecrawl npx \
  -a @anthropic/firecrawl-mcp \
  -e FIRECRAWL_API_KEY=fc-xxx

# Register a Python MCP server
panopticon register my-tools python \
  -a -m \
  -a my_mcp_server
```

<Callout type="warn">
Environment variables in `servers.yaml` are merged with the current process environment. Never store production secrets directly in this file. Use a secrets manager or reference environment variables that are set at the system level.
</Callout>

## Guardrail Policy

Smith's guardrail enforcement is configured at `~/.panopticon/guardrails.yaml`. This controls which tools are allowed, what gets logged, and how aggressively results are scanned.

```yaml title="~/.panopticon/guardrails.yaml"
level: MEDIUM
blocked_tools:
  - "dangerous_server.delete_all"
  - "admin.drop_database"
context_budget_warn: 0.85
context_budget_compress: 0.90
context_budget_emergency: 0.95
```

### Guardrail Options

<TypeTable
  type={{
    level: { type: '"HIGH" | "MEDIUM" | "LOW" | "MINIMAL"', description: 'Enforcement level controlling validation depth', default: '"MEDIUM"' },
    blocked_tools: { type: 'string[]', description: 'Tool names or server.tool patterns to block unconditionally', default: '[]' },
    context_budget_warn: { type: 'float', description: 'Context window usage fraction triggering a warning', default: '0.85' },
    context_budget_compress: { type: 'float', description: 'Context window usage fraction triggering compression', default: '0.90' },
    context_budget_emergency: { type: 'float', description: 'Context window usage fraction triggering emergency measures', default: '0.95' },
  }}
/>

### Enforcement Levels

| Level | Behavior |
|-------|----------|
| **HIGH** | Block dangerous + HIGH-only tools. Validate all parameters. Scan all results for secrets. Inject steering context. |
| **MEDIUM** | Block dangerous tools. Scan results when flagged. Log tool calls. |
| **LOW** | Log tool calls only. No blocking or scanning. |
| **MINIMAL** | Complete pass-through. No logging, blocking, or scanning. |

### Always-Blocked Tools

These tools are blocked regardless of enforcement level:

| Tool | Reason |
|------|--------|
| `rm_rf` | Destructive filesystem operation |
| `drop_database` | Destructive database operation |
| `format_disk` | Destructive storage operation |

### HIGH-Only Tools

These tools require `level: HIGH` to execute:

| Tool | Reason |
|------|--------|
| `git_push_force` | Destructive git operation |
| `delete_branch` | Removes code branches |
| `modify_permissions` | Changes access control |

## Gemini Key Pool

API keys are configured via environment variables (not YAML files, to avoid storing secrets on disk).

### Environment Variables

<TypeTable
  type={{
    GEMINI_API_KEYS: { type: 'string', description: 'Comma-separated regular Google AI Studio API keys for round-robin rotation' },
    VERTEX_API_KEYS: { type: 'string', description: 'Comma-separated Vertex AI Express API keys (auto-prefixed with "vertex:")' },
    GEMINI_API_KEY: { type: 'string', description: 'Single key fallback (used when GEMINI_API_KEYS is not set)' },
  }}
/>

### Resolution Order

1. `GEMINI_API_KEYS` (comma-separated) + `VERTEX_API_KEYS` (auto-prefixed)
2. `GEMINI_API_KEY` (single key)
3. Error: `KeyPoolError` if no keys found

### Pool Parameters

The key pool has internal defaults that are not currently exposed via configuration files:

| Parameter | Default | Description |
|-----------|---------|-------------|
| `cooldown_seconds` | 60.0 | How long to sideline a rate-limited key |
| `rpm_limit` | 10 | Max requests per minute per key before proactive rotation |

<Callout type="info">
The RPM limit of 10 per key is set conservatively because Panopticon operations (video upload, visual search) are heavier than typical Gemini calls. If you have high-tier API keys with higher rate limits, you can adjust this in code at `panopticon/keys.py`.
</Callout>

### Docker Compose Infrastructure

The `docker-compose.yml` in the repository root provides a single-node Temporal deployment:

```yaml title="docker-compose.yml"
services:
  postgresql:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: temporal
      POSTGRES_USER: temporal
      POSTGRES_DB: temporal
    volumes:
      - temporal-pg-data:/var/lib/postgresql/data

  temporal:
    image: temporalio/auto-setup:latest
    depends_on:
      postgresql:
        condition: service_healthy
    environment:
      - DB=postgres12
      - DB_PORT=5432
      - POSTGRES_USER=temporal
      - POSTGRES_PWD=temporal
      - POSTGRES_SEEDS=postgresql
    ports:
      - "7233:7233"   # gRPC frontend

  temporal-ui:
    image: temporalio/ui:latest
    depends_on:
      - temporal
    environment:
      - TEMPORAL_ADDRESS=temporal:7233
    ports:
      - "8233:8080"   # Web UI

volumes:
  temporal-pg-data:
```

## Memory System Paths

The memory system uses fixed paths under `~/.panopticon/memory/`. These are not currently configurable via environment variables.

| File | Purpose | Format |
|------|---------|--------|
| `~/.panopticon/memory/episodic.jsonl` | Episodic memory records | JSONL |
| `~/.panopticon/memory/hive.jsonl` | Hive mind observations | JSONL |
| `~/.panopticon/memory/promotions.jsonl` | Promoted patterns | JSONL |
| `~/.panopticon/memory/traces.jsonl` | Archived execution traces | JSONL |
| `~/.cortex/lightning/traces/` | Agent Lightning trace files | JSONL per day |
| `~/.cortex/lightning/rollouts/` | RL training rollouts | JSONL per day |
| `~/.cortex/lightning/state.json` | Training state | JSON |

## Agent Integration Paths

| Path | Purpose |
|------|---------|
| `~/.claude/agents/oracle.md` | Oracle agent definition |
| `~/.claude/agents/smith.md` | Smith agent definition |
| `~/.claude/agents/trinity.md` | Trinity agent definition |
| `~/Zulip/bots/oracle-zuliprc` | Oracle's Zulip credentials |
| `~/.cortex/trinity/` | Trinity optimization hints |
| `~/.notebooklm-mcp/auth.json` | NotebookLM authentication cache |

## Validation

Panopticon validates server configurations when loading `servers.yaml`. A server entry is valid if it has a non-empty `command` field. Invalid entries are silently skipped during schema discovery.

The guardrail policy defaults to `MEDIUM` if `guardrails.yaml` is missing or unparseable. The catalog defaults to an empty state if `catalog.json` is missing.

<Callout type="info">
There is no `panopticon config validate` command yet. Configuration errors surface at runtime when a specific feature is used. The health check (`panopticon health`) is the best way to verify your configuration is working correctly.
</Callout>
